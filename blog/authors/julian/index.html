From Engine to Hylozoa Engine

June 3, 2026 · 3 min read
Julian
DevOps and Quality Assurance Engineer

The project has been ongoing since since March 2025, let's review its current state together:

runs smoother outside of a gif​

Overview​

Hylozoa-Engine runs on a ECS and currently supports the following components

Features/Components​

Prefabs​

Users can save an entity config as a .json to instanciate it whenever they need.

Lua scripting​

Users can write lua scripts to control their entity.
The following methods are currently exposed:

  • instanciate() to instanciate an entity from a prefab
  • destroy() to destroy an entity
  • get_transform() to get the transform of an entity
  • is_key_pressed() to check if the user pressed a key

The standard math library is also exposed.
And the following entrypoint is read from the script by the game engine:

  • onUpdate(entity, deltatime) logic to run every frame

Planned improvements​

  • A way to communication between entities' scripts to coordinate them
  • Lua entrypoint for the following events:
    • Collision detection
    • Entity emotion change

Renderable​

Entity can have a sprite or a colored shape attached to them.

Planned improvements​

The entity's sprite will be animated.

Physics simulation​

Entities with a RigidBody will be affected by gravity and collides with other collider components.

Noise listener​

Entity (even the camera) will hear the noises comming from the scene depending on the distance from the source.

From Engine to Hylozoa Engine​

Hylozoa Engine has a solid foundation, but it lacks any features that helps to build complex ecosystems.

Therefore, the following updates are planned for the comming year:

Prefab parenting​

Following evolution, species have inherited traits from their parent species.
As such, using prefabs parenting would allow the users to mimic this aspect of evolution.

Example:
Ape | Monkey |-> Jungle Monkey
|-> Beach Monkey

AND Crab -> Beach Monkey

BeachMonkey.prefab.json:

{
    "prefab" : {
        "parents": [
            "Monkey",
            "Crab"
        ]
    },
    "components" : ...
}

Such that (Ape < Monkey < Crab) + beach monkey fields == Beach Monkey.

Proposal:
We simply add a prefab.parent field to the prefab's json.
We then construct the child's prefab by parsing every parent and overwriting over the previous parent's fields.

Senses components​

View​

We don't support vision yet.
This will be done using simple raycasts.
The following methods will be provided to scripts:

OnSightEnter(); -- 1st frame
OnSightLeave(); -- 1st frame of not seeing after being seen
OnSight(); -- All other frames

Scripting​

Listing surrounding entities​

Example: an entity should know how many friend or foe surrounds it to make fight-or-flight decision.

Proposal: PerceptionMemory component

The PerceptionMemory component make use of all senses that are attached to the entity as components (hear, see, smell). Internally, it will store every entity perceived and their time of perception whenever the senses triggers.
The method get_perception-list() : {entity} will be available to retrieve a list of entity perceived in the last memoryTime seconds.

Ground rework​

Currently, creating ground is a bit of a hassle.

Proposal: Constructing terrain from a .svg.
That method has already proven itself in games such as Super Mario Wonder.
This would be stored in the scene's configuration and not as an entity. Colors can be configured to act as different materials (dirt, sand, water, ...).

Default assets/prefabs​

To ease the user experience, default prefabs should be included.

Proposal:

  • Default hylozoa mascot entity with all perception components active.
  • Default random man that is player controlled.
  • A pixar like funny ball to show physics.
  • Some default ground (might be scene config or prefab).

Measuring and Optimizing Technical Performance

March 1, 2025 · 3 min read
Julian
DevOps and Quality Assurance Engineer

Find here our plan regarding Hylozoa's testing and optimizing goals.​

📏 Metrics

Our current goal is to track the following 3 metrics:​

Memory usage​

The following sub-metrics will be tracked

  • Current Memory
  • Peak Memory
  • Memory Budget
  • Allocation count

in 4 scenes:

  • Empty scene.
  • Beach demo scene.
  • A Stress scene that is overloaded with entities.
  • A streming scene that is constantly loading and unloading entities.

For a duration of 10 seconds.

CI will fail is a commit introduce an unjustified spike in any of these metrics.

CPU Usage​

We will track these sub-metrics:

  • Frame Time
    • Average
    • P95
    • P99
  • Thread Time

By including the professional level tool Tracy Profiler into the Hylozoa source code. We will then integrate Intel VTune at a later date deeper profiling.

CPU miscache​

We will first track these sub-metrics:

  • L1
  • L2
  • LLC/L3

Using perf stat -d hylozoa. This metric too will be improved by the Intel VTune integration.

Tracy Profiler​

"Tracy is a live profiler for C++ applications that shows frame timings, thread activity, memory allocations, and CPU hotspots in real time. It combines manually instrumented zones with statistical sampling to provide both precise timing information and broader performance analysis."

ChatGPT when asked to explain the Tracy Profiler's github readme tagline​

It offers access to many profiler features by simply including the library and using keywords unto existing code. These features include:

  • Thread Visualization
  • GPU Profiling
  • Memory Tracking
  • Frame Analysis

It is very effective in CI tests.

It is a high level tools that works best when paired with lower level tools such as:

Intel VTune​

Intel VTune is a low level CPU & GPU performance analysis tool. It's main purpose is to investigate and understand performance issues.

It profiles:

  • Cache misses
  • Branch prediction
  • Pipeline stalls

It would not be used directly in CI pipeline.
